3.Cluster模式（数据分片)

Cluster模式是Redis的一种高级集群模式，它通过数据分片和分布式存储实现了负载均衡和高可用性。在Cluster模式下，Redis将所有的键值对数据分散在多个节点上。每个节点负责一部分数据，称为槽位。通过对数据的分片，Cluster模式可以突破单节点的内存限制，实现更大规模的数据存储

Redis Cluster将数据分为16384个槽位，每个节点负责管理一部分槽位。当客户端向Redis Cluster发送请求时，Cluster会根据键的哈希值将请求路由到相应的节点。具体来说，Redis Cluster使用CRC16算法计算键的哈希值，然后对16384取模，得到槽位编号

主备和主从的区别

主备模式和主从模式是在计算机系统、数据库管理等领域常用的两种数据冗余和高可用性策略，它们有以下区别：

一、概念定义

  • 主备模式
    • 主备模式是一种简单的冗余配置方式。在这种模式下，有一个主系统（节点）处于活跃状态，负责处理所有的业务操作。而备用系统（节点）平时处于等待状态，它会不断地监测主系统的状态。
    • 例如，在一个网络服务器的主备模式中，主服务器正常提供网页服务，接收用户的访问请求并进行处理。备用服务器则定期向主服务器发送心跳信号来检查其是否正常运行，自身并不处理用户请求。
  • 主从模式
    • 主从模式则是一种存在主节点和从节点分工协作的工作方式。主节点主要负责接收和处理写操作（如数据的插入、更新、删除），而从节点主要负责处理读操作。
    • 以数据库系统为例，在主从数据库架构中，当有用户向主数据库插入一条新的记录（写操作）时，主数据库执行插入操作后，会将更新的数据同步到从数据库。用户对数据的查询（读操作）可以由从数据库来处理，这样可以分担主数据库的负载。

二、数据同步方式

  • 主备模式
    • 数据同步通常是单向的，从主系统到备用系统。当主系统出现故障时，备用系统需要确保能够获取主系统最后的完整数据状态，才能接替主系统的工作。
    • 比如，有些主备服务器采用日志传输的方式进行同步。主服务器将操作日志（如文件系统的操作记录、数据库事务日志等）不断地发送给备用服务器，备用服务器根据这些日志来更新自己的数据状态，以保持和主服务器的数据一致。
  • 主从模式
    • 数据同步方式相对复杂，需要确保从节点的数据与主节点的数据保持一致。同步可以是基于事务的同步或者是基于数据块的同步。
    • 例如，在数据库的主从复制中，主数据库在执行一个写事务后，会将这个事务的内容（如 SQL 语句或者事务日志中的修改记录）发送给从数据库。从数据库接收到这些信息后，按照相同的顺序在自己的系统中重新执行这些操作，从而实现数据同步。

三、故障切换过程

  • 主备模式
    • 当主系统发生故障时，备用系统会检测到这个情况并立即启动，接替主系统进行工作。这个切换过程相对简单直接，因为备用系统在正常情况下只是处于备份状态，没有其他复杂的操作。
    • 不过，故障切换过程需要确保备用系统能够快速且正确地接管主系统的所有资源，包括网络地址、存储设备等。例如，在一些网络应用的主备切换中，备用服务器需要快速获取主服务器的 IP 地址，这样用户的请求才能顺利地被备用服务器接收和处理。
  • 主从模式
    • 故障切换相对复杂。如果主节点发生故障，需要选择一个从节点来升级为主节点，这个从节点要能够承担主节点的写操作任务。而且，在切换过程中，要确保从节点的数据是最新的，并且在切换后，新的主节点和其他从节点之间的数据同步能够正常进行。
    • 比如，在一个分布式数据库系统中，主节点故障后，系统需要根据一定的规则（如从节点的数据更新程度、从节点的性能等）来选择一个合适的从节点作为新的主节点。然后，需要对整个系统的数据同步机制进行调整，以适应新的主从架构。

四、应用场景

  • 主备模式
    • 适用于对数据一致性要求极高，且读操作相对较少的场景。例如，一些关键的金融交易系统的核心服务器，平时主要是进行交易处理（写操作），这些系统对数据的准确性和系统的可靠性要求非常高，采用主备模式可以确保在主服务器出现故障时，备用服务器能够无缝接管，避免数据丢失和业务中断。
  • 主从模式
    • 适用于读操作频繁、写操作相对较少的场景，如一些大型的内容分发网站。这些网站的用户主要是进行数据查询（读操作），通过设置多个从节点来处理读操作，可以有效地减轻主节点的负载，提高系统的整体性能和响应速度。同时，主从模式也能够在一定程度上保证数据的冗余和高可用性。

分布式系统的脑裂问题

是指在主从集群中，同时有两个主节点，它们都能接收写请求。而脑裂最直接的影响，就是客户端不知道应该往哪个主节点写入数据，结果就是不同的客户端会往不同的主节点上写入数据。而且严重的话，脑裂会进一步导致数据丢失

产生2个主节点的原因是原本的主节点和部分节点之间由于网路故障，或者延时抖动导致心跳机制出现问题，其他节点由于检测不到主节点的心跳认为它已经挂了，其实并没有挂掉，其他节点就会重新选举新的主节点，这样集群中就会有2个主节点。

Raft 算法： 为了避免分布式系统中的脑裂问题，通常要求候选节点需要获得超过一半（即 <font style="color:#000000;">(eligible 节点数量 + 1) / 2</font>）的选票才能成为主节点。因此，**eligible**** 节点的数量最好是奇数**，以确保选票的总数为奇数，这样可以避免出现无法形成多数的情况（比如 50% 对 50% 的平票问题）
